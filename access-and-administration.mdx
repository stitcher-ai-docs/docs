This page covers how you get into StitcherAI and what you can touch — tenancy, identity, access, and the surfaces you work in.

## Tenancy

StitcherAI is organized in three levels you work with directly:

<Steps>
  <Step title="Organization">
    Your company — the top-level container, tied to your identity provider.
  </Step>
  <Step title="Environment">
    An isolated workspace within the organization (for example, one per business unit). Data and configuration in one environment are **inaccessible from another**, even within the same organization.
  </Step>
  <Step title="Pipeline">
    A cost model. An environment can hold one or more.
  </Step>
</Steps>

## Identity and access

<CardGroup cols={2}>
  <Card title="Single sign-on" icon="key">
    Sign in through your identity provider (OpenID Connect). The same identity is used for the web app and the VS Code editor.
  </Card>

  <Card title="Environment-scoped roles" icon="user-lock">
    Role-based access granted per environment, least-privilege by default. The full security model is in [Security, tenancy & AI guardrails](/security).
  </Card>
</CardGroup>

<Note>
  Each pipeline's configuration lives in a **GitHub repository you control.** Repository access, branch-protection rules, and pull-request reviews are managed in GitHub — so your existing engineering access controls govern changes to the cost model.
</Note>

## Connections and credentials

<Warning>
  Data-source credentials are entered and managed in the web app — **never** in your configuration repository — and are validated before use. Setting up a specific connection is covered per provider in the data-foundation documentation.
