A **Complex Field Unpack** operation explodes keys from an **object / map** column — for example a provider `Tags` or `SkuPriceDetails` field — into their own columns. Each key you select becomes a separate column you can then filter, map, or attribute on.

For JSON stored as text in a column, use [JSON Field Unpack](/operator-guides/json-field-unpack) instead — the key selection rules work identically.

**Availability.** Runs in both **Prepare** and **Enrich**. In Prepare it flattens raw provider blobs before FOCUS exists; in Enrich it works on FOCUS-era data.

<Note>
  Choosing the operation (Step 1) and defining [scope](/pipeline-overview#building-blocks-shared-across-stages) (Step 2) are identical across every operator. This guide covers only **Step 3: Configure field unpack**. Unlike column-writing operators, field unpack has **no merge policy** — each selected key is written to its own target column.
</Note>

## Configure the unpack

<Steps>
  <Step title="Pick the source column">
    Set **Source column name** to the complex column to unpack — a Map or struct field such as `Tags` or `SkuPriceDetails`.
  </Step>
  <Step title="Add one or more key selection rules">
    Each rule's **Selection type** decides which keys are pulled out:

    - **Select a single key** — one key by its exact, case-sensitive name (**Key value**).
    - **Select a list of keys** — several keys by exact name (**Key values**); each becomes its own column.
    - **Select with a key pattern** — every key whose name matches a regular expression (**Key pattern**, e.g. `^label_.*`); each match becomes its own column.
  </Step>
  <Step title="Name the target column(s)">
    Set the rule's **Target configuration**:

    - **Column type** — FOCUS or Custom (custom columns start with `x_`).
    - For a single key, a **Target column name**. For a list or a pattern, a **Target column prefix** and **Target column suffix** that wrap each key name — e.g. key `id` with prefix `condition_` and suffix `_foo` becomes `condition_id_foo`.
    - Optionally **Enable target column transformation** (derive the value with an expression) or **Enable target column default value** (used when the key is absent).
  </Step>
</Steps>

## Worked example — flatten a `team` tag

Extract the `team` key from a provider `Tags` map into its own attribution column:

- **Source column name:** `Tags`
- **Selection type:** Select a single key · **Key value:** `team`
- **Column type:** Custom · **Target column name:** `x_team`
- **Enable target column default value:** `unassigned` (for rows whose tag is missing)

Every row's `team` tag now lives in its own `x_team` column, ready to attribute on.

## Result & gotchas

- Key names are **exact and case-sensitive** for single key and list modes — use a **key pattern** to catch variants by regular expression.
- With a **list** or **pattern**, each matched key becomes its own column; the prefix and suffix build the column name (`prefix` \+ key \+ `suffix`).
- There's **no merge policy** on field unpack — pick target names that don't collide with columns already in the dataset.
