A run executes your cost model end to end: it connects to your sources, normalizes and attributes the data, builds artifacts, evaluates governance, and delivers output. This page covers the three ways to run it, how to watch a run, and how to fix one that fails.

## The three run types

They differ only in **scope**, **where output goes**, and **whether governance and notifications fire**.

<CardGroup cols={3}>
  <Card title="Standard run" icon="play">
    The full model — all stages, all sources, whole-month periods. Governance and notifications fire; output goes to **all** destinations.
  </Card>

  <Card title="Scheduled run" icon="clock">
    A hands-off standard run on a schedule you define (typically daily). Normal production.
  </Card>

  <Card title="Debug run" icon="flask">
    A scoped, low-cost run to validate a change. Writes to a **debug** destination; notifications suppressed.
  </Card>
</CardGroup>

### Standard run

The full model: all stages, all configured sources, and whole-month periods (the current month can be month-to-date). Governance is evaluated, notifications fire, and output is written to every configured [destination](/engage). Use it to **backfill or update** data — and note that a schedule invokes exactly this kind of run.

<Warning>
  A standard run updates the production data your organization relies on. Run it deliberately — for routine production, prefer a scheduled run.
</Warning>

### Scheduled run

A standard run on a schedule you define, left to run unattended — this is normal production operation. It runs the full scope at each interval and takes no extra inputs; a scheduled run can't be named or put into debug mode, because the point is that it runs the same full model every time without anyone starting it.

### Debug run

A scoped, low-cost run for validating a change **before it ships**. You narrow it three ways and run it against the branch you're working on (see [Authoring & version control](/authoring-and-version-control)):

<Steps>
  <Step title="Limit the period to a single day">
    Pick one day within a month rather than the whole month, so the run is cheap and fast.
  </Step>
  <Step title="Narrow the data scope">
    Run one provider, or a single data source within a provider. Business data still processes, because downstream steps depend on it.
  </Step>
  <Step title="Run only the stages you're changing">
    Stop at a chosen stage, or resume from a recent debug session so you skip the stages you've already run. Start _at_ the stage you're editing so your change actually executes.
  </Step>
</Steps>

A debug run writes to a **debug destination** rather than production, and **notifications are suppressed** — so you can iterate freely without touching live data or alerting anyone. Debug runs can be named, but can't be scheduled.

## Watching a run

Open a run on the **run details page** to watch it live.

<Frame caption="The run details page: per-stage progress and failure logs">
  ![The run details page: per-stage progress and failure logs](/images/fig-16-run-details.png)
</Frame>

<Note>
  A pipeline runs **one run at a time**. Runs can be **named** — so a backfill is easy to find later — and **cancelled** from the run details page if you need to stop one.
</Note>

The page shows **per-stage and per-step progress**: which stage is running, which steps have completed, and which is in flight. (For what each stage does, see [The semantic pipeline](/pipeline-overview).)

## Diagnosing a failure

When a step fails, the run details page is also where you fix it.

<Steps>
  <Step title="Open the failed step">
    The failed step is marked, with its **error and logs** shown directly. Read the reason there — you don't go hunting for it.
  </Step>
  <Step title="Fix the configuration">
    Make the change in the [VS Code editor](/authoring-and-version-control) on your working branch.
  </Step>
  <Step title="Re-run a scoped debug run">
    Re-run on just the relevant scope — the single day, the one source, the stages you changed — to confirm the fix cheaply.
  </Step>
  <Step title="Promote when it's clean">
    Once the debug run passes, merge your change and let a standard run (or the schedule) update production.
  </Step>
</Steps>

<Tip>
  A recent debug session keeps the work already done, so a follow-up debug run can **start from a stage you've already passed** instead of re-running everything. Iterate on the stage you're actually changing.
</Tip>
