StitcherAI normalizes every cost source — a cloud bill, a SaaS invoice, a Kubernetes export — into one common language: **FOCUS**, the [FinOps Open Cost and Usage Specification](https://focus.finops.org). Because everything speaks FOCUS, cost from any provider is directly comparable, and the output is yours.

## One language, open output

<CardGroup cols={2}>
  <Card title="A common schema" icon="layer-group">
    Every source lands in the same FOCUS columns, so a Kubernetes export, a SaaS invoice, and a cloud bill can be compared and combined directly.
  </Card>

  <Card title="Open formats you own" icon="box-open">
    Outputs are **Parquet and CSV** — portable and warehouse-native, not a proprietary API. They land in your own storage.
  </Card>
</CardGroup>

<Note>
  FOCUS is an open standard, not a StitcherAI format. StitcherAI's founder co-created it. Building your finance practice on FOCUS means your data stays portable and isn't locked to any one tool.
</Note>

## Guaranteed vs best-effort columns

StitcherAI is precise about what normalization delivers. Some columns are always populated; others depend on what the source actually contains.

<Tabs>
  <Tab title="Guaranteed">
    A core set is populated for **every** normalized source:

    - Billing and charge periods (start and end)
    - The cost metrics — billed cost and effective cost
    - Provider and account
    - Service
    - Currency
    - Charge category

    This core is what makes any two sources comparable on day one.
  </Tab>
  <Tab title="Best-effort">
    Richer columns are populated **to the extent the source data supports them**:

    - Resource ids, names, and types
    - Regions
    - Tags and labels
    - Commitment and discount detail
    - Unit prices and quantities
    - SKUs

    Where a source carries this detail, you get it; where it doesn't, the column stays empty rather than being guessed.
  </Tab>
</Tabs>

<Warning>
  **StitcherAI does not manufacture definition that isn't present in the source.** If a vendor's data doesn't distinguish regions or carry resource ids, those columns won't be invented. This is what keeps the model trustworthy — it never implies a level of detail the underlying data can't support. Where richer definition is needed for AI spend, [telemetry](/managing-ai-spend) supplies it.
</Warning>
