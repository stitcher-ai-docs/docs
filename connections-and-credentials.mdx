title: "Connections & credentials"
description: "How to define a data connection in StitcherAI, why credentials live in the web app and never in your configuration repository, how connections are validated before use, and where to find ready-made per-provider setup."
---

A connection tells StitcherAI where your data lives and how to reach it. You define connections in the web app, where credentials are stored securely and validated before anything depends on them.

## Credentials live in the web app

<Warning>
  Data-source credentials are entered and managed in the web app — **never** in your configuration repository. Your cost model refers to a connection only by its **name**; the actual keys and roles are held in StitcherAI. Nothing sensitive is ever committed alongside the model.
</Warning>

This separation is what lets the cost model itself be version-controlled and peer-reviewed (see [Authoring & version control](/access-and-administration)) without ever exposing a secret in a pull request. When a run executes, the connection name is resolved to the stored credentials at that moment.

## Define and validate a connection

<Steps>
  <Step title="Provision provider access">
    Create the least-privilege access the provider needs — a role, service account, or API key. Use the ready-made infrastructure below so you don't have to assemble permissions by hand.
  </Step>
  <Step title="Create the connection in the web app">
    Enter the location and credentials. They're stored in StitcherAI, referenced from your model by name only.
  </Step>
  <Step title="Validate before use">
    StitcherAI checks the connection against the provider to confirm it can actually read — or, for a destination, write — the data.
  </Step>
  <Step title="Craft sources and destinations from it">
    With the base connection in place, craft a full [data source or destination](/data-foundation) from it without re-entering connection details.
  </Step>
</Steps>

<Info>
  Validation runs **when you create the connection**, not when a run later fails. On success you're cleared to use it; on failure StitcherAI reports a clear, actionable reason — for example, a missing storage permission or an expired key — so you can fix the access and revalidate.
</Info>

## Per-provider setup, ready to use

Each provider has its own access requirements. StitcherAI publishes detailed, copy-pasteable infrastructure — least-privilege roles, service accounts, and storage grants — in a public Terraform-modules repository, so you provision exactly what each connector needs and nothing more.

<Card title="StitcherAI Terraform modules" icon="github" href="https://github.com/StitcherAI/tf-modules">
  Per-provider setup modules — provision the access, then reference the connection in StitcherAI.
</Card>

What each family of providers needs, at a glance:

<Tabs>
  <Tab title="Cloud storage & warehouse">
    - **AWS** — a cross-account role with read access to the cost-export bucket, plus an external id.
    - **Azure** — an app identity with storage roles scoped to the export path.
    - **Google Cloud** — a service account with read access to BigQuery and / or the storage bucket.
    - **Snowflake** — a user, role, and warehouse with read access to the usage views.
  </Tab>
  <Tab title="Provider APIs">
    - **OpenAI · Anthropic / Claude · Twilio · Confluent · Elastic · MongoDB** — an API key or token with permission to read usage and cost.

    The exact scope per provider is documented in its module in the [tf-modules repository](https://github.com/StitcherAI/tf-modules).
  </Tab>
  <Tab title="File & PDF sources">
    - **Any vendor invoice or custom cost file** — read access to the storage location (S3, GCS, or Azure Storage) where the files land.

    For how unstructured files become FOCUS, see [AI-powered ingestion](/ai-ingestion).
  </Tab>
</Tabs>

<Tip>
  Pushing data from your own side? Customer-side sync tools write to **your** storage with **your** keys, and a StitcherAI file connection then reads from that location — your bucket credentials are never given to StitcherAI. The connection you define here is the read side of that handoff.
</Tip>
